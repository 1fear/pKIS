# Roadmap TakSklad

Этот roadmap описывает практичные направления развития TakSklad как складской WMS-экосистемы. Это не строгий release plan, а рабочий ориентир для выбора изменений, которые реально помогают складу.

## 0. Стратегический Фокус

На 29.05.2026 принято решение готовить TakSklad к переходу на VDS/серверную архитектуру.

- Desktop-версия поддерживается как рабочий складской инструмент, но без крупных новых улучшений и глубокого рефакторинга ради самой desktop-версии.
- В desktop исправляются только критичные вещи: сканирование, сохранение, импорт, печать, обновления и понятные ошибки.
- Новые фоновые процессы проектируются для backend/workers: Telegram, SkladBot, отчёты, общий state, синхронизация с Google Sheets.
- При выборе между временной desktop-доработкой и подготовкой серверной архитектуры приоритет у VDS, если складская работа не заблокирована.

## 1. Усилить Локальное Приложение

Приоритет: высокий.

- Сохранять стабильное и предсказуемое сканирование.
- Делать ошибки Google Sheets восстановимыми через `pending_saves.json`.
- Поддерживать надежную печать сводных листов на Windows и fallback через `lp` на macOS/Linux.
- Улучшать логирование дублей КИЗов и критических ошибок.
- Сохранять локальные backup-файлы для восстановления.
- Не хранить секреты и локальные настройки в Git.
- Делать проверку обновлений понятной и безопасной.
- Показывать ошибки так, чтобы сотрудник склада понимал следующий шаг.

## 2. Улучшить Telegram-Управление

Приоритет: высокий.

- Сохранять безопасную и полезную отправку отчетов.
- Учитывать Google Sheets и локальные `pending_saves` в отчете "сканы за сегодня".
- Показывать документы по импорту с прогрессом, например `davron.xlsx | 20/30`.
- Принимать Excel-файлы в Telegram и запускать тот же импорт, что делает desktop-приложение.
- Блокировать повторный импорт уже успешно загруженного Excel-файла по хэшу.
- Отправлять Excel-отчеты по импорту с листами "Сводка", "Позиции", "КИЗы" и "Недосканировано".
- Отправлять критические ошибки вместе с сегодняшним логом.
- Уведомлять пользователей после успешного импорта.
- Добавлять только те Telegram-действия, которые уменьшают ручную работу.

## 3. Централизовать Данные

Приоритет: средний.

- Хранить историю импортов в структурированном и надежном виде.
- Отделить историю сканов от временных очередей.
- Сохранять прогресс по каждому документу.
- Отслеживать дубли и недосканированные позиции.
- Хранить историю формирования отчетов.
- Сохранять достаточно данных для ответа на вопросы кто, что и когда сделал.

## 4. Подготовить Backend

Приоритет: средний.

Статус на 30.05.2026: начат backend MVP-каркас. Уже добавлены FastAPI shell, `GET /health`, стартовые API-контракты, PostgreSQL-схема, Dockerfile, VDS Docker Compose и тесты структуры. Это ещё не рабочий серверный релиз: desktop не подключён к backend, CRUD/миграции/воркеры не реализованы.

Backend становится полезным, когда:

- Telegram-бот должен работать 24/7;
- несколько складских компьютеров должны работать с одним источником данных;
- отчеты и мониторинг должны работать без открытого desktop-приложения;
- руководителям нужен удаленный доступ к текущему состоянию склада.

Возможные задачи backend:

- реализовать центральный API для импортов и сканов поверх Postgres;
- добавить Alembic/миграционную процедуру для изменений схемы;
- подключить desktop через feature flag: сначала dual-write, затем чтение заказов из backend;
- вынести Telegram bot в отдельный worker;
- вынести SkladBot sync в отдельный worker;
- делать отчеты по расписанию на сервере;
- хранить backup-файлы и проверить restore;
- вести аудит действий;
- добавить пользователей и роли;
- оставить Google Sheets как экспорт/витрину, а не основной источник истины.

## 5. Добавить Веб-Интерфейс

Приоритет: позже.

Полезные разделы веб-панели:

- статус текущих документов;
- прогресс сканирования;
- недосканированные КИЗы;
- история импортов;
- отчеты;
- логи и ошибки;
- настройки;
- пользователи и роли;
- админ-инструменты.

Веб-интерфейс должен быть рабочим и компактным. Его задача - помогать сотрудникам и руководителям быстро принимать решения, а не выглядеть как рекламный сайт.

## 6. Добавить Аналитику Процессов

Приоритет: позже.

Аналитика должна отвечать на практические вопросы:

- какие документы еще не завершены;
- где замедляется сканирование;
- какие товары или клиенты чаще создают ошибки;
- сколько КИЗов сканируется за смену;
- как часто падает Google Sheets или печать;
- что сейчас находится в очередях и требует внимания.

## Главный Принцип

Каждое изменение должно делать складскую работу проще, быстрее, прозрачнее или надежнее. Не стоит добавлять сложность, если она прямо не помогает этим целям.
