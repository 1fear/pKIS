# TakSklad: Полное Описание Функционала

Документ описывает текущее состояние проекта TakSklad на версии `1.1.17` по коду в репозитории на `26.05.2026`.

Документ не содержит реальные токены Telegram, JWT, приватные ключи Google credentials или другие секреты. Идентификаторы таблиц и service account описаны без конкретных значений; технические имена листов и файлов указаны как рабочая инфраструктура проекта.

## 1. Назначение Системы

TakSklad - локальное desktop-приложение для складской работы с заказами, КИЗами, Google Sheets, Excel-документами, печатью, отчётами, Telegram-ботом и автообновлением.

Основная роль приложения:

- принять Excel-документы с заказами;
- нормализовать строки заказов;
- записать данные в Google Sheets;
- показать складу активные задания;
- принять КИЗы со сканера;
- защитить от повторного сканирования КИЗов;
- сохранить КИЗы в Google Sheets;
- восстановить работу при сбоях сети, Google Sheets, печати или Telegram;
- напечатать сводный лист;
- сформировать Excel-отчёты;
- отправить отчёты, логи, backup-файлы и документы через Telegram;
- принудительно обновляться через GitHub Releases.

TakSklad сейчас является локальной точкой управления складом. Telegram-бот и автоотчёты работают только пока приложение запущено на компьютере склада.

## 2. Общая Архитектура

Проект разделён на модули:

- `src/taksklad/main.py` - интерфейс Tkinter, сканирование, печать, отчёты, Telegram, автообновление, очереди и обработка ошибок.
- `src/taksklad/config.py` - константы приложения, имена листов, имена файлов, версия, настройки колонок, цвета UI, размеры печати.
- `src/taksklad/excel_import.py` - импорт Excel, нормализация строк, дедупликация, подготовка записей для Google Sheets.
- `excel_normalizer.py` - поиск структуры произвольного Excel-шаблона, алиасы колонок, определение даты из файла/контекста, пропуск итоговых строк.
- `src/taksklad/sheets.py` - подключение к Google Sheets, структура листов, запись импортов, чтение активных заказов, запись КИЗов.
- `orders.py` - ID заказов, ключи дублей, статус заказа, группировка заказов.
- `catalog.py` - справочник товаров, количество штук в блоке, расчёт плановых блоков.
- `src/taksklad/storage.py` - общий JSON-файл данных, миграция старых JSON-файлов, чтение и запись секций.
- `skladbot.py` - API-клиент SkladBot, получение заявок, нормализация заявок и сравнение с заказами.
- `skladbot_sync.py` - синхронизация номеров заявок SkladBot в лист `data`.
- `geocoding.py` - получение адреса по координатам через Яндекс Геокодер.
- `utils.py` - нормализация текста, дат, чисел, хэши, работа с кодами и именами файлов.

## 3. Основной Рабочий Процесс

Типовой процесс склада:

1. Пользователь запускает `TakSklad.exe`.
2. Приложение проверяет автообновление.
3. Приложение подключается к Google Sheets.
4. Активные заказы загружаются из листа `data`.
5. Пользователь импортирует Excel через приложение или отправляет Excel в Telegram.
6. Импорт записывает строки только в лист `data`; лист `Данные` больше не используется приложением.
7. Склад выбирает заказ из списка.
8. Оператор сканирует КИЗы по позиции.
9. После достижения плана по позиции КИЗы записываются в Google Sheets.
10. После завершения всех позиций по клиенту/оплате/адресу печатается сводный лист.
11. В конце дня формируется Excel-отчёт.
12. Отчёты и логи могут отправляться в Telegram.

## 4. Google Sheets

Приложение работает с одной Google-таблицей по `SPREADSHEET_ID`.

Текущая рабочая таблица:

- название: `TakSklad рабочая база`;
- владелец: рабочий Google-аккаунт организации;
- `SPREADSHEET_ID`: актуальный идентификатор рабочей Google-таблицы TakSklad из локальной конфигурации;
- доступ для приложения: актуальный service account TakSklad с ролью **Editor**;
- старый service account удалён из доступа после переноса.

Рабочие листы текущей таблицы:

- `data` - главный рабочий лист приложения для склада, импортов, SkladBot-номеров и сканирования.
- `Архив` - лист для будущего/ручного хранения архивных записей.
- `Возвраты` - лист для будущего/ручного хранения возвратов.

Лист `Данные` в текущей рабочей таблице отсутствует. Импорт в этот лист полностью удалён из приложения: кнопка импорта и Telegram-бот пишут только в `data`.

### 4.1. Лист `data`

`data` - главный рабочий лист для TakSklad.

Рабочие колонки:

- `Дата отгрузки`
- `Тип оплаты`
- `Клиент`
- `Адрес`
- `Торговый представитель`
- `Товары`
- `Кол-во ШТ`
- `Кол-во блок`
- `Отсканированные коды`
- `Статус`

Служебные колонки пишутся с позиции `AA`:

- `ID заказа`
- `ID импорта`
- `Источник файла`
- `Строка файла`
- `Дата импорта`
- `Номер заявки SkladBot`
- `ID заявки SkladBot`
- `Статус SkladBot`
- `Последняя проверка SkladBot`

Приложение поддерживает старое имя колонки `Дата получения заказа` как совместимость, но новый рабочий заголовок - `Дата отгрузки`.

При запуске и импорте приложение проверяет структуру листа:

- если нужных служебных колонок нет, они добавляются;
- если служебные колонки есть в старых местах, данные мигрируются в служебный блок начиная с `AA`;
- обязательные колонки валидируются перед чтением и записью.

### 4.2. Лист `Архив`

`Архив` сейчас перенесён в новую таблицу как отдельный рабочий лист для будущего процесса архивации.

На текущем этапе приложение не переносит выполненные строки в `Архив` автоматически. Выполнение заказа определяется колонкой `Статус` в `data`, а активный список приложения строится только по невыполненным строкам.

### 4.3. Лист `Возвраты`

`Возвраты` сейчас перенесён в новую таблицу как отдельный рабочий лист для будущего процесса возвратов.

На текущем этапе основная логика сканирования КИЗов и дневных отчётов работает с листом `data`. Возвраты как отдельный бизнес-процесс ещё не автоматизированы внутри TakSklad.

### 4.4. Исторический Лист `Данные`

Ранее импорт сначала писал строки в лист `Данные`, а потом в `data`.

Старая логика листа `Данные` была такой:

- первая дата - текущая дата импорта;
- вторая дата - дата доставки/отгрузки из Excel;
- торговый представитель;
- клиент;
- ИНН;
- адрес;
- координаты;
- товар;
- тип оплаты;
- тип лида/статус лида;
- количество штук;
- количество блоков;
- остальные видимые поля до `X` заполнялись пустыми значениями, если данных нет.

Перед записью новых строк приложение копировало оформление предыдущей строки: шрифт, размер, заливку, границы, выравнивания, числовые и датовые форматы.

Сейчас этот лист не используется. Код импорта больше не открывает, не проверяет и не заполняет лист `Данные`.

## 5. Импорт Excel

Импорт доступен двумя способами:

- через кнопку `ИМПОРТ EXCEL` в приложении;
- через отправку Excel-файла в Telegram-бот.

Поддерживаемые форматы:

- `.xlsx`
- `.xlsm`

Если в Excel есть лист `Заявки`, приложение проверяет его первым. Если листа `Заявки` нет, нормализатор ищет подходящий лист и строку заголовков по первым строкам книги.

Начиная с гибкого нормализатора шаблонов, файл не обязан быть строго в одном фиксированном формате. Приложение пытается найти обязательные колонки по алиасам и может принять отчёты вида `Конструктор отчетов`, если в них есть клиент, оплата, товар и количество.

### 5.3. Нормализация Строк Excel

Для каждой строки Excel приложение:

- пропускает полностью пустые строки;
- проверяет, что заполнены клиент, оплата, товар и количество;
- приводит дату к формату `дд.мм.гггг`;
- если даты нет, ставит текущую дату;
- читает адрес;
- если адрес пустой, но есть координаты, пытается получить адрес через Яндекс Геокодер;
- если адрес не удалось получить, пишет адрес как `Координаты: ...` и добавляет предупреждение;
- если нет ни адреса, ни координат, ставит `Адрес не указан`;
- считает план блоков через справочник товаров;
- сохраняет исходный файл, строку файла и дату импорта;
- формирует `ID импорта` и `ID заказа`.

### 5.4. Справочник Товаров Во Время Импорта

При расчёте блоков приложение использует `product_catalog`.

Для каждого товара хранится:

- название;
- количество штук в одном блоке;
- флаг `requires_kiz`.

Если товара ещё нет в справочнике, он создаётся автоматически с дефолтным размером блока `10` штук.

Если количество штук не делится на размер блока, план блоков округляется вверх. Например, если в блоке `10` штук, а в Excel `21`, план будет `3` блока, и приложение добавит предупреждение.

Текущий нюанс: флаг `requires_kiz` сохраняется в справочнике, но рабочая логика сканирования сейчас всё равно ожидает КИЗы по позиции. То есть настройка есть, но она ещё не меняет поведение сканирования.

### 5.5. Дедупликация При Импорте

Начиная с `1.1.6`, приложение не складывает похожие строки.

Главное правило:

> Если строка отличается хотя бы одним значимым полем, она остаётся отдельной строкой. Если строка полностью совпадает, она считается дублем и пропускается.

Точный дубль внутри выбранных Excel-файлов определяется по полям:

- дата;
- тип оплаты;
- клиент;
- адрес;
- торговый представитель;
- ИНН;
- товар;
- количество;
- координаты;
- исходный адрес;
- тип лида.

Дата входит в ключ дубля. Поэтому одинаковый клиент, товар, адрес и количество на разные даты - это разные заказы.

Пример, это дубль:

| Дата | Клиент | Товар | Кол-во |
|---|---|---|---|
| 19.05.2026 | ИП Даврон | Chapman Gold | 20 |
| 19.05.2026 | ИП Даврон | Chapman Gold | 20 |

Пример, это не дубль:

| Дата | Клиент | Товар | Кол-во |
|---|---|---|---|
| 19.05.2026 | ИП Даврон | Chapman Gold | 20 |
| 19.05.2026 | ИП Даврон | Chapman Gold | 30 |

Пример, это тоже не дубль:

| Дата | Клиент | Товар | Кол-во |
|---|---|---|---|
| 19.05.2026 | ИП Даврон | Chapman Gold | 20 |
| 20.05.2026 | ИП Даврон | Chapman Gold | 20 |

### 5.6. Проверка Дублей В Уже Загруженных Таблицах

Перед записью приложение проверяет, нет ли позиции уже в `data`.

Для `data` используются:

- `ID импорта`;
- `ID заказа`;
- ключ по доступным полям рабочего листа: дата, тип оплаты, клиент, адрес, торговый представитель, товар, количество.

Важно: текущий рабочий лист `data` не хранит ИНН, координаты и тип лида как основные рабочие поля. Поэтому повторная проверка уже существующих строк в `data` сравнивает только поля, которые реально есть в `data`.

Для листа `Данные` отдельная проверка дублей больше не выполняется, потому что приложение больше не пишет в этот лист.

### 5.7. Порядок Записи При Импорте

Порядок строк сохраняется.

Алгоритм:

1. Excel читается сверху вниз.
2. Полные дубли пропускаются.
3. Все остальные строки остаются в исходной последовательности.
4. Строки записываются в `data` в том же порядке.
5. Лист `Данные` не заполняется; весь импорт завершается записью только в `data`.

Если выбрано несколько Excel-файлов, порядок идёт по порядку, который вернул диалог выбора файлов, а внутри каждого файла - сверху вниз.

### 5.8. Предпросмотр Импорта В Приложении

Перед записью приложение показывает:

- сколько файлов проверено;
- сколько строк найдено;
- сколько новых позиций после проверки;
- сколько клиентов;
- сколько товаров;
- общее количество штук;
- план блоков;
- сколько адресов получено из координат;
- сколько координат осталось без адреса;
- сколько повторных строк в Excel пропущено;
- сколько повторных позиций уже найдено в таблице;
- ошибки и предупреждения по файлам.

После предпросмотра пользователь подтверждает загрузку в Google Sheets.

### 5.9. Импорт Через Telegram

Если авторизованный пользователь отправляет Excel-файл в бот:

1. бот принимает файл только если приложение запущено;
2. файл скачивается во временный файл;
3. файл проверяется по SHA256 на повторный успешный импорт;
4. запускается тот же импорт, что и через кнопку приложения;
5. если новых позиций нет, бот сообщает причину;
6. если импорт успешен, бот отправляет сообщение с результатом;
7. после импорта документ появляется в разделе `Документы по импорту`.

Повторный импорт через Telegram блокируется по хэшу файла, если этот файл уже был успешно загружен раньше.

## 6. Загрузка И Группировка Заказов

При запуске и при нажатии `ОБНОВИТЬ` приложение читает `data`.

Приложение:

- проверяет обязательные колонки;
- обновляет статус строк, если строка уже фактически выполнена;
- берёт активные заказы;
- собирает все уже существующие КИЗы из Google Sheets;
- добавляет КИЗы из локальной очереди `pending_saves`;
- синхронизирует очередь `pending_saves`, если Google Sheets снова доступен.
- синхронизирует номера заявок SkladBot, если SkladBot настроен.

Заказы группируются для интерфейса по ключу:

- номер заявки SkladBot;
- клиент;
- тип оплаты;
- адрес.

Это значит, что один и тот же клиент на один адрес, но с разными типами оплаты, будет показан разными заказами. Номер заявки SkladBot дополнительно разделяет группы и выводится первым в списке.

Внутри выбранной группы позиции сортируются по номеру строки Google Sheets.

### 6.1. SkladBot-Номера Заявок

TakSklad умеет подтягивать номера заявок SkladBot в формате `WH-R-...`/`WR-...` в лист `data`.

Рабочая идея:

- Excel импортируется в `data`;
- приложение сравнивает строки `data` с заявками SkladBot;
- если найдено ровно одно совпадение, номер заявки записывается в колонки SkladBot;
- если совпадений несколько, ставится статус `Несколько совпадений`;
- если совпадений нет, ставится статус `Не найдено`;
- если совпадение SkladBot не найдено или SkladBot временно недоступен, приложение всё равно показывает активные невыполненные Google-заказы для КИЗов.

Колонки SkladBot в `data`:

- `Номер заявки SkladBot`;
- `ID заявки SkladBot`;
- `Статус SkladBot`;
- `Последняя проверка SkladBot`.

Критерии сравнения заказа из `data` и заявки SkladBot:

- дата отгрузки в TakSklad должна совпадать с датой выгрузки в SkladBot;
- клиент/получатель сравнивается после нормализации кавычек, регистра, тире и лишних пробелов;
- тип оплаты берётся из комментария заявки SkladBot и сравнивается без учёта регистра;
- адрес используется только как мягкая дополнительная проверка, он не должен быть главным блокирующим критерием;
- товар сравнивается по нормализованному SKU: цвет `brown`/`red`/`gold` и формат `OP`/`SSL`;
- количество сравнивается только в блоках по каждому товару.

Важно: адрес не должен отсеивать заявку SkladBot, если совпали дата, клиент, тип оплаты, товары и блоки. Адрес нужен для диагностики и ручной проверки, но не как жёсткий ключ.

Правило количества:

- Excel может приходить в пачках/штуках;
- SkladBot работает в блоках;
- в `data` должны сохраняться оба значения: `Кол-во ШТ` и `Кол-во блок`;
- для SkladBot-сравнения используется только `Кол-во блок`.

Правило цены:

- базовая цена блока Chapman для текущего процесса - `240 000 сум`;
- если импортный Excel содержит сумму, она сохраняется;
- если сумма не указана, TakSklad считает её автоматически по формуле `Кол-во блок * 240 000`;
- в сводочном листе для водителя/экспедитора должна выводиться общая сумма заказа к получению.

Правило логистического отчёта:

- отчёт логистики не зависит от склада, КИЗов, закрытия смены или архива;
- менеджер формирует его из Telegram по выбранной дате отгрузки;
- в логистический шаблон должны попадать координаты доставки, а не адрес;
- если TakSklad получает адрес через геокодер, адрес используется для удобства склада и сверок, но не заменяет координаты в логистическом отчёте.
- если координат нет, логистический отчёт не должен формироваться молча с пустым маршрутом.

Фильтр заявок SkladBot:

- клиент SkladBot: `ООО Bastion Import Chapman MCHJ`;
- тип заявки: `3PL отгрузка`;
- активные заявки;
- завершённые заявки за последние 2 дня, чтобы недавно закрытые заявки ещё могли подтянуть номер.

Обновление SkladBot выполняется:

- фоново примерно каждые 10 минут, пока приложение открыто и не занято текущим заказом;
- отдельно от кнопки `ОБНОВИТЬ`, чтобы загрузка списка для КИЗов не ждала SkladBot;
- после фоновой записи номеров приложение может перечитать список, если оператор не находится внутри заказа.

Если SkladBot настроен и синхронизация прошла без ошибок, найденные номера заявок выводятся в списке и участвуют в группировке. Заказы без найденного номера SkladBot не скрываются: если они активные и не выполнены, они остаются видимыми в списке для КИЗов.

## 8. Сканирование КИЗов

Оператор выбирает заказ и сканирует коды в поле сканирования.

Проверка КИЗа:

- код не пустой;
- начинается с `01`;
- длина не меньше `20` символов;
- длина не больше `120` символов;
- нет русских букв;
- нет пробелов или переносов;
- символы входят в допустимый набор ASCII с поддержкой GS-разделителя `\x1d`;
- код не сканировался в текущей позиции;
- код не найден в уже загруженных Google Sheets;
- код не найден в локальной очереди записи;
- код не найден среди завершённых в текущей сессии позиций.

После принятия каждого кода:

- код сразу пишется в локальный backup;
- код добавляется в текущий список;
- прогресс позиции обновляется;
- поле сканирования очищается;
- фокус возвращается в поле сканирования.

Если план позиции достигнут, кнопка `СЛЕДУЮЩАЯ ПОЗИЦИЯ` становится активной.

## 9. Отмена Последнего Сканирования

Кнопка `ОТМЕНИТЬ ПОСЛЕДНИЙ КОД` удаляет только последний код, который ещё не был записан в Google Sheets.

Нельзя отменить:

- код, если нет активной позиции;
- код, если список сканов пуст;
- код, который уже сохранён в Google Sheets.

Отмена также записывается в локальный backup.

## 10. Сохранение КИЗов В Google Sheets

После выполнения позиции пользователь нажимает `СЛЕДУЮЩАЯ ПОЗИЦИЯ`.

Приложение:

1. проверяет, что количество сканов равно плану блоков;
2. ищет строку заказа в `data`;
3. сначала пробует `_row_number`;
4. если `_row_number` устарел, ищет похожую строку по данным заказа;
5. проверяет, что в строке нет других уже записанных КИЗов;
6. проверяет, что эти КИЗы не встречаются в других строках;
7. записывает `Отсканированные коды`;
8. обновляет `Статус`;
9. пишет backup успешной записи.

Статус считается так:

- если план блоков больше нуля и количество КИЗов больше или равно плану, статус `Выполнено`;
- иначе статус `Не выполнено`.

## 11. Очередь Записи `pending_saves`

Если Google Sheets временно недоступен, приложение может сохранить позицию локально.

В очередь записи попадает:

- данные заказа;
- список КИЗов;
- причина ошибки;
- время создания;
- время обновления.

При следующем обновлении списка приложение пытается отправить очередь в Google Sheets.

Если отправка успешна:

- запись удаляется из очереди;
- пишется backup `pending_save_synced`;
- список заказов перечитывается.

КИЗы из `pending_saves` учитываются как занятые, чтобы оператор не мог отсканировать их повторно.

Текущий нюанс: функция определения retryable-ошибок достаточно широкая. Часть неизвестных ошибок может попасть в очередь как временная проблема.

## 12. Backup-Файлы Сканирования

Каждое важное действие сканирования пишется в `scan_backups/scan_backup_дд.мм.гггг.jsonl`.

В backup попадают:

- время;
- действие;
- строка Google Sheets;
- дата заказа;
- клиент;
- адрес;
- товар;
- тип оплаты;
- один код или список кодов.

Типы действий:

- `scan` - принят новый код;
- `undo_scan` - отменён последний код;
- `position_saved` - позиция записана в Google Sheets;
- `position_queued` - позиция сохранена локально из-за сбоя записи;
- `pending_save_synced` - очередь записи успешно синхронизирована;
- `address_finished` - заказ по адресу завершён и сводка напечатана.

## 13. Завершение Заказа

Заказ в интерфейсе - это группа:

- клиент;
- тип оплаты;
- адрес.

Чтобы завершить заказ:

1. все позиции группы должны быть завершены;
2. у приложения должен быть список завершённых позиций;
3. пользователь подтверждает параметры печати;
4. приложение добавляет сводку в очередь печати;
5. приложение печатает сводку;
6. если печать успешна, сводка удаляется из очереди;
7. пишется backup завершения адреса;
8. заказ убирается из текущего списка.

Если печать не удалась, запись остаётся в очереди `pending_prints`.

## 14. Печать Сводных Листов

Сводный лист создаётся как PNG-картинка.

Параметры:

- размер этикетки `100 x 100 мм`;
- DPI `203`;
- до 3 товаров на страницу;
- если товаров больше 3, создаётся несколько страниц;
- внизу печатается дата и время.

В сводке выводятся:

- заголовок;
- адрес;
- клиент;
- торговый представитель;
- номер страницы;
- список товаров;
- количество блоков;
- количество штук;
- итоги по странице.

Печать:

- на Windows используется PowerShell и `System.Drawing.Printing`;
- на macOS/Linux используется команда `lp`;
- можно указать имя принтера;
- если имя принтера не указано или стоит `Термопринтер`, используется принтер по умолчанию.

Параметры печати сохраняются в `print_settings`.

## 15. Очередь Печати `pending_prints`

Перед печатью сводки приложение добавляет её в очередь.

Если печать успешна, запись удаляется.

Если приложение закрылось или печать упала, запись остаётся.

При следующем запуске приложение проверяет очередь и предлагает напечатать непечатанные сводки.

В очереди хранится:

- ID сводки;
- время создания;
- адрес;
- список товаров;
- КИЗы по позициям.

## 16. Справочник Товаров

Позволяет:

Если товара нет, он добавляется автоматически при импорте.

## 18. Отчёт За День

Кнопка `ЗАВЕРШИТЬ ДЕНЬ (ОТЧЁТ)` формирует Excel-файл по сканам за текущую дату.

В отчёт попадают:

- КИЗы из Google Sheets;
- КИЗы из локальной очереди `pending_saves`, если они относятся к текущему дню и ещё не попали в Google Sheets.

Листы отчёта:

- `Терминал`;
- `Перечисление`;
- `Не распознано`, если есть такие строки.

Каждая строка отчёта соответствует одному КИЗу.

Поля отчёта:

- клиент;
- торговый представитель;
- адрес;
- товар;
- тип оплаты;
- штук в блоке;
- количество блоков;
- итого штук;
- КИЗ.

После формирования отчёта приложение может отправить его в Telegram, если это включено в настройках.

### 18.1. Автоотправка Дневного Отчёта

Дневной отчёт не должен сбрасываться при закрытии приложения в течение дня. Актуальность отчёта определяется текущей датой и данными в Google Sheets/локальных очередях.

Логика:

- если пользователь завершает день вручную, отчёт за текущий день формируется и отправляется в Telegram;
- если день не завершили вручную, приложение пытается отправить отчёт автоматически в конце дня;
- стандартное время автоотправки: `23:55`;
- если приложение не работало в конце дня, при следующем запуске оно проверяет пропущенные дни и отправляет неотправленные дневные отчёты;
- состояние отправки хранится в `daily_report_state` внутри `TakSklad_data.json`;
- в отчёт включаются данные Google Sheets и локальная очередь `pending_saves`.

Это нужно, чтобы закрытие приложения оператором не обнуляло дневную картину и не терялись сканы, принятые в течение дня.

## 19. Документы По Импорту

Для импортированных документов приложение хранит в `data`:

- источник файла;
- строку файла;
- дату импорта;
- ID импорта.

Telegram-раздел `Документы по импорту` строится по этим данным.

Для каждого документа показывается:

- имя файла;
- дата импорта;
- количество позиций;
- выполненные позиции;
- план КИЗ;
- отсканированные КИЗы;
- КИЗы в локальной очереди;
- прогресс вида `20/30`.

По документу можно скачать Excel-отчёт.

Листы отчёта по документу:

- `Сводка`;
- `Позиции`;
- `КИЗы`;
- `Недосканировано`.

В отчёт попадают и КИЗы из Google Sheets, и КИЗы из локальной очереди `pending_saves`.

## 20. Telegram-Бот

Telegram работает как удалённый канал управления, пока локальное приложение запущено.

Настройки хранятся в `telegram_settings` внутри общего файла данных или в legacy-файле `telegram_settings.json`.

Основные настройки:

- `enabled` - включён ли Telegram;
- `bot_token` - токен бота;
- `chat_id` - старый формат одного чата;
- `chat_ids` - список разрешённых получателей;
- `send_reports` - отправлять отчёты;
- `send_scan_backups` - отправлять backup сканов;
- `send_pending_files` - отправлять очереди;
- `send_error_log` - отправлять лог ошибок.

Реальные токены и chat IDs не должны попадать в Git.

### 20.1. Авторизация Telegram

Команды принимаются только от разрешённых `chat_ids`.

Если сообщение или callback пришли от неизвестного chat ID, приложение:

- не выполняет команду;
- пишет предупреждение в лог.

### 20.2. Меню Telegram

Бот показывает inline-кнопки:

- `Скачать сканы за сегодня`;
- `Документы по импорту`;
- `Скачать сегодняшний лог`.

В разделе документов кнопки имеют вид:

```text
имя_файла.xlsx | отсканировано/план
```

### 20.3. Скачать Сканы За Сегодня

Команда формирует Excel-отчёт за текущий день и отправляет файл в чат.

Если сканов нет, бот отправляет текстовое сообщение.

### 20.4. Документы По Импорту

Команда показывает последние импортированные документы.

По кнопке документа бот формирует Excel-отчёт именно по этому документу и отправляет его в чат.

### 20.5. Скачать Сегодняшний Лог

Команда собирает записи из `TakSklad.log` только за текущую дату и отправляет `.txt` файл.

Если ошибок за день нет, файл всё равно создаётся с информационной строкой.

### 20.6. Импорт Excel Через Telegram

Если пользователь отправляет Excel-файл:

- проверяется доступ пользователя;
- проверяется формат `.xlsx` или `.xlsm`;
- файл скачивается через Telegram API;
- проверяется повторный импорт по SHA256;
- выполняется импорт в Google Sheets;
- бот отправляет результат;
- при ошибке бот отправляет сообщение и сегодняшний лог.

Если приложение занято другой операцией, бот просит отправить файл повторно позже.

### 20.7. Очередь Telegram `pending_telegram`

Если Telegram временно недоступен, документы могут попасть в очередь.

В очереди хранится:

- путь файла;
- подпись;
- причина ошибки;
- количество попыток;
- время создания и обновления.

При запуске и далее в фоне приложение пытается отправить очередь повторно.

### 20.8. Защита От Отправки Секретов

Функция безопасной отправки файлов запрещает отправлять:

- `credentials.json`;
- `telegram_settings.json`;
- `TakSklad_data.json`;
- файл ключа Яндекс Геокодера.

Это сделано, чтобы бот не отправил приватные настройки и токены.

## 21. Ошибки, Логи И Уведомления

Основной лог:

- `TakSklad.log`

В лог пишутся:

- ошибки Google Sheets;
- ошибки Telegram;
- ошибки импорта;
- ошибки печати;

- критические ошибки интерфейса;
- сбои автообновления;

При критической ошибке приложение:

- показывает окно ошибки;
- пишет traceback в лог;

Если найден дубль КИЗа, приложение пытается определить:

- строку Google Sheets;
- дату;
- тип оплаты;
- клиента;
- адрес;
- торгового представителя;
- товар;
- количество;
- план блоков;
- статус;
- сколько кодов уже в строке.

Эти детали пишутся в лог и отправляются в Telegram.

## 22. Автообновление

TakSklad обновляется через GitHub Releases.

Компоненты:

- `APP_VERSION` в `src/taksklad/config.py`;
- публичный `version.json` в ветке `main`;
- GitHub Release с `TakSklad-windows-x64.zip`;
- переходный `TakSklad.exe` для старых onefile-версий;
- SHA256 checksum;
- PowerShell/batch-скрипт установки обновления.

При запуске приложение:

1. читает `version.json`;
2. сравнивает `APP_VERSION` с `latest_version`;
3. сравнивает `APP_VERSION` с `min_supported_version`;
4. если нужна новая версия, блокирует рабочие кнопки;
5. скачивает новый ZIP-пакет папочной сборки;
6. проверяет SHA256;
7. запускает временный `.ps1`;
8. закрывает текущее приложение;
9. PowerShell-скрипт обновляет `TakSklad.exe` и `_internal`;
10. запускает новую версию.

Начиная с `1.1.7`, основная сборка Windows делается через PyInstaller `--onedir`.
Это уменьшает риск ошибок вида `Failed to load Python DLL` после автообновления,
потому что приложение больше не стартует из временной `_MEI`-папки.

Папочная сборка поставляется как `TakSklad-windows-x64.zip`.
Обновление распаковывает архив во временную папку, заменяет исполняемые файлы
и `_internal`, но не трогает локальные рабочие данные:

- `TakSklad_data.json`;
- `credentials.json`;

## 26. Безопасность И Секреты

Секреты не должны попадать в репозиторий:

- Telegram bot token;
- Google service account JSON;
- JWT/API-токены сторонних сервисов;
- приватные chat IDs, если они не нужны в публичной документации;
- реальные backup-файлы;
- реальные отчёты;
- логи с персональными или коммерческими данными.

Telegram умеет отправлять рабочие файлы, но блокирует отправку основных файлов с секретами.

## 27. Google Credentials

Для доступа к Google Sheets используется service account.

Приложение ищет credentials так:

1. секция `credentials` в `TakSklad_data.json`;
2. файл `credentials.json` рядом с приложением.

Требуются права:

- доступ к Google Sheets;
- доступ к Google Drive API;
- права на нужную Google-таблицу.

Если credentials не найдены, приложение показывает ошибку при запуске.

Текущая схема доступа:

- Google-таблица принадлежит рабочему аккаунту организации;
- приложение работает через актуальный service account TakSklad;
- service account имеет роль редактора в таблице;
- старый service account был нужен только для переноса листов и после переноса удалён из доступа.

## 28. Геокодирование

Если в Excel нет адреса, но есть координаты, приложение пытается получить адрес через Яндекс Геокодер.

Поведение:

- если адрес получен, он используется как адрес заказа;
- если адрес не получен, адрес становится `Координаты: ...`;
- добавляется предупреждение по строке;
- результаты координат кешируются в рамках одного импорта, чтобы не делать повторные запросы.

Ключ Яндекс Геокодера должен храниться локально или в настройке, а не публиковаться.

## 29. Статусы Заказов

Статус в `data` хранится в колонке `Статус`.

Основные значения:

- `Не выполнено`;
- `Выполнено`.

Приложение считает позицию выполненной, если:

- план блоков больше `0`;
- количество отсканированных КИЗов больше или равно плану блоков.

Если статус пустой, приложение может заполнить его автоматически.

Если статус похож на выполненный, строка считается неактивной.

Если статус похож на невыполненный или кодов меньше плана, строка остаётся активной.

## 30. Контроль Дублей КИЗов

Дубли КИЗов проверяются на нескольких уровнях:

- внутри текущей позиции;
- во всех уже существующих кодах Google Sheets;
- в локальной очереди `pending_saves`;
- в завершённых позициях текущей сессии;
- при записи в Google Sheets дополнительно проверяются другие строки таблицы.

Если дубль найден в Google Sheets, приложение формирует подробности и отправляет их в лог/Telegram.

## 31. Ограничения Текущей Версии

Текущие известные особенности:

- Telegram-бот не является отдельным сервером и работает только пока открыто локальное приложение.
- Если старый exe не может скачать обновление из-за SSL, нужен один ручной переход на `1.1.6` или новее.
- Текущая рабочая таблица содержит только `data`, `Архив`, `Возвраты`. Лист `Данные` больше не используется приложением.
- `data` хранит меньше полей, чем исходный Excel, поэтому повторная проверка дублей использует только поля, доступные в `data`.
- Флаг `Нужен КИЗ` в справочнике сохраняется, но ещё не отключает обязательное сканирование.
- Приложение рассчитано на один активный операторский процесс на компьютере. Несколько параллельных копий могут конфликтовать по локальным очередям и файлам.
- Очереди локальные для конкретного компьютера. Если несколько складских ПК работают с одной таблицей, централизованной очереди пока нет.
- Telegram polling идёт из desktop-приложения, поэтому 24/7-команды требуют отдельного backend/worker в будущем.
- SkladBot-сравнение зависит от полноты данных в заявке. Если в SkladBot несколько заявок совпадают по дате, клиенту, оплате, адресу и товарам/количеству, приложение не выбирает одну наугад, а ставит статус `Несколько совпадений`.
- Если SkladBot не нашёл номер заявки или временно недоступен, складская работа не блокируется: приложение показывает активные невыполненные строки из Google Sheets без обязательного фильтра по SkladBot-номеру.

## 32. Что Уже Покрывает Стабильность

Приложение защищает рабочий процесс так:

- локальный backup создаётся при каждом принятом КИЗе;
- КИЗ не принимается, если backup не удалось записать;
- при временном сбое Google Sheets позиция сохраняется в `pending_saves`;
- при сбое печати сводка остаётся в `pending_prints`;
- при сбое Telegram-файлы остаются в `pending_telegram`;
- отчёты учитывают `pending_saves`;
- дубли КИЗов логируются с деталями;
- ошибки отправляются в Telegram при настроенном боте;
- автообновление проверяет SHA256 скачанного exe;
- HTTPS-запросы используют встроенный CA-bundle.

## 33. Практический Чеклист Работы Склада

Перед началом:

1. Запустить `TakSklad.exe`.
2. Дождаться обновления списка.
3. Если предложено обновление, дать приложению обновиться.
4. Проверить, что нет ошибок Google credentials.
5. Проверить, что принтер доступен.
6. При необходимости открыть `КОНТРОЛЬ`.

Импорт:

1. Нажать `ИМПОРТ EXCEL` или отправить Excel в Telegram.
2. Проверить предпросмотр.
3. Убедиться, что количество строк, дублей и предупреждений выглядит нормально.
4. Подтвердить импорт.
5. Убедиться, что документ появился в списке заказов и в Telegram-разделе документов.

Сканирование:

1. Выбрать заказ.
2. Сканировать КИЗы по текущей позиции.
3. После выполнения плана нажать `СЛЕДУЮЩАЯ ПОЗИЦИЯ`.
4. Повторить для всех позиций группы.
5. Подтвердить печать сводки.

Конец дня:

1. Проверить очередь записи в статистике.
2. Нажать `ЗАВЕРШИТЬ ДЕНЬ (ОТЧЁТ)`.
3. Проверить, что отчёт сформирован.
4. Проверить Telegram, если отправка отчётов включена.
