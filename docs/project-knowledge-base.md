# TakSklad: База Знаний Проекта

Актуально на 31.05.2026. Документ собирает рабочую картину проекта: что уже есть, как устроены данные, какие решения приняты, где узкие места и что важно помнить при дальнейшей разработке.

Документ намеренно не содержит Telegram token, Google private key, реальные chat_id, приватные ключи, пароли VPS или другие секреты. Секреты должны жить только в локальных файлах, переменных окружения или защищённом хранилище.

## 0.1 Зафиксированные Правила Chapman-Процесса

Эти правила закрывают риски по Smartup/Excel, SkladBot, адресам и логистике. Для MVP 2.0 они считаются утверждёнными.

- Smartup/Excel не обязан содержать отдельный файл или поле даты отгрузки. Менеджер сам выбирает актуальную дату отгрузки в Telegram перед импортом.
- Для сопоставления со SkladBot количество сравнивается только в блоках. Excel может приходить в пачках/штуках, но перед SkladBot-матчингом всё приводится к `Кол-во блок`.
- Товар сравнивается по нормализованному Chapman SKU: цвет `brown`/`red`/`gold` и формат `OP`/`SSL`. Длинные хвосты названия в SkladBot не должны создавать отдельный товар.
- Адрес не является жёстким критерием SkladBot-матчинга. Он помогает диагностике, но совпадение держится на дате отгрузки/выгрузки, клиенте, оплате, товаре и блоках.
- В логистический отчёт должны попадать координаты доставки. Адрес может храниться для склада и сверки, но не заменяет координаты в логистическом Excel.
- Цена блока Chapman для текущего процесса - `240 000 сум`. Если импорт не содержит готовую сумму, TakSklad считает её по блокам.

## 1. Что Такое TakSklad

TakSklad - локальное desktop-приложение для складской работы:

- импортирует Excel-заказы;
- нормализует строки и пишет их в Google Sheets;
- показывает активные позиции для сканирования КИЗов;
- принимает ввод со сканера;
- сохраняет сканы в Google Sheets;
- печатает сводные листы;
- делает дневные отчёты;
- работает с Telegram-ботом;
- подтягивает номера заявок SkladBot;
- обновляется через GitHub Releases;
- хранит локальные очереди восстановления при сбоях.

Сейчас это не полноценный backend/WMS, а локальная точка управления складом. Но направление развития - складская экосистема: desktop + backend + PostgreSQL + Telegram worker + SkladBot worker + веб-панель + аналитика.

## 3. Рабочие Данные

Основной источник рабочих заказов сейчас - Google Sheets.

Рабочие листы:

- `data` - главный лист для заказов, импортов, SkladBot-номеров и сканов;
- `Архив` - зарезервирован под будущий архивный процесс;
- `Возвраты` - зарезервирован под возвраты;
- `_TakSklad_System` - служебный лист для Telegram-lock и общего Telegram state.

Legacy-лист `Данные` больше не используется. Импорт из UI и Telegram должен писать только в `data`.

Ключевые колонки `data`:

- `Дата отгрузки`;
- `Тип оплаты`;
- `Клиент`;
- `Адрес`;
- `Торговый представитель`;
- `Товары`;
- `Кол-во ШТ`;
- `Кол-во блок`;
- `Отсканированные коды`;
- `Статус`;
- `Номер заявки SkladBot`;
- `ID заявки SkladBot`;
- `Статус SkladBot`;
- `Последняя проверка SkladBot`.

Служебные поля импорта располагаются начиная с `AA`, чтобы не ломать рабочую часть таблицы.

## 5. Основной Складской Процесс

Типовой процесс:

1. Оператор запускает `TakSklad.exe`.
2. Приложение проверяет версию и рабочие файлы.
3. Читает активные строки из Google Sheets.
4. Показывает группы заказов.
5. Фоново подтягивает номера SkladBot.
6. Оператор выбирает заказ.
7. Сканирует КИЗы по текущей позиции.
8. Каждый принятый КИЗ сначала попадает в локальный backup.
9. После завершения позиции/заказа сканы пишутся в Google Sheets.
10. Если Google недоступен, запись попадает в `pending_saves`.
11. После завершения заказа печатается сводный лист.
12. В конце дня формируется отчёт и отправляется в Telegram, если включено.

Главный принцип: сканирование не должно зависеть от SkladBot или Telegram. Если внешняя интеграция падает, складская работа продолжается через локальные backup/очереди.

## 6. Группировка Заказов

Заказы группируются по рабочим признакам:

- номер заявки SkladBot, если найден;
- клиент;
- тип оплаты;
- адрес.

Это защищает от смешивания:

- одного клиента с разными типами оплаты;
- одного клиента на разные адреса;
- разных SkladBot-заявок одного клиента.

Заказы без номера SkladBot не скрываются, если они активные и не выполнены. Они показываются как `Без номера SkladBot`.

## 7. SkladBot: Текущая Логика

SkladBot нужен для подтягивания номеров заявок вида `WH-R-...` / `WR-...`.

Что используется:

- `GET /requests` - список заявок;
- `GET /requests/show/{id}` - карточка заявки;
- клиент SkladBot: `ООО Bastion Import Chapman MCHJ`;
- целевой тип заявки: `3PL отгрузка`.

Важная особенность API: в списке `/requests` есть `created_at`, но нет рабочей `Дата выгрузки`. `Дата выгрузки` появляется только в карточке заявки.

Текущая схема:

1. Получить до `SKLADBOT_REQUESTS_LIMIT` заявок.
2. Первично отфильтровать по `created_at` за окно `SKLADBOT_SYNC_LOOKBACK_DAYS`.
3. Открыть карточки оставшихся заявок.
4. В карточке взять `unloading_date`, клиента, адрес, комментарий, товары.
5. Сравнить с группами заказов из `data`.
6. Если найдено ровно одно совпадение - записать номер и ID заявки.
7. Если совпадений несколько - поставить `Несколько совпадений`.
8. Если совпадений нет - поставить `Не найдено`.

Почему окно не 1 день: SkladBot-заявки часто создаются за несколько дней до отгрузки. Поэтому окно по `created_at` должно покрывать обычный логистический цикл. Сейчас используется 14 дней.

Критерии матчинга:

- дата `data.Дата отгрузки` строго равна `SkladBot.unloading_date`;
- клиент равен после нормализации названия компании;
- тип оплаты сравнивается с комментарием заявки;
- адрес используется как мягкий диагностический признак и не должен блокировать совпадение сам по себе;
- товары сравниваются по нормализованному цвету/формату Chapman, количество - только в блоках;
- если остаётся несколько кандидатов - случайный выбор запрещён.

Нормализация клиента:

- регистр не важен;
- кавычки, типографские кавычки, точки, дефисы и пунктуация не ломают совпадение;
- разные слова всё ещё считаются разными клиентами.

Известная недоработка: если в Google попала только часть товаров заявки, а в SkladBot карточка содержит полный набор товаров, текущий строгий product-match может поставить `Не найдено`. Для этого нужен безопасный partial-match.

## 9. Telegram

Telegram сейчас работает внутри desktop-приложения, а не как отдельный сервер.

Функции:

- скачать сканы за сегодня;
- посмотреть документы по импортам;
- скачать сегодняшний лог;
- импортировать Excel-файл;
- получить результат импорта;
- получить критические ошибки;
- получить отчёты.

Авторизация:

- команды разрешены только из whitelisted `chat_ids`;
- неизвестные chat_id логируются и игнорируются;
- реальные chat_id и bot token не должны попадать в git.

Проблема двух компьютеров:

- если оба вызывают `getUpdates`, Telegram отдаёт `HTTP 409 Conflict`;
- добавлен Google Sheets lock `telegram_poll`;
- владелец lock слушает Telegram;
- второй компьютер пропускает polling и пробует позже.

Дополнительная защита:

- общий `last_update_id` хранится в `_TakSklad_System`;
- это защищает от двойной обработки одного Excel-файла при гонке lock;
- локальный `telegram_state` остаётся fallback, если Google недоступен.

Временность решения: Telegram-lock в Google Sheets - рабочий компромисс, но не финальная архитектура. Правильное решение - отдельный Telegram worker на сервере.

## 10. Google Sheets

Google Sheets сейчас является и рабочей таблицей, и частью coordination layer.

Используется для:

- хранения заказов;
- записи сканов;
- статусов;
- SkladBot-номеров;
- Telegram lock/state;
- частично как operational database.

Проблемы:

- много `get_all_values`;
- несколько компьютеров увеличивают read quota;
- Telegram lock/state тоже расходует read/write quota;
- при `429 quota exceeded` lock может не освободиться;
- при timeout могут не пройти фоновые SkladBot-синхронизации.

Краткосрочные решения:

- уменьшать количество чтений;
- объединять чтение листа там, где возможно;
- backoff/retry на `429`;
- реже обновлять Telegram lock;
- не запускать несколько локальных экземпляров на одном компьютере;
- не делать SkladBot sync слишком часто.

Долгосрочное решение: вынести operational storage в PostgreSQL, а Google Sheets оставить как внешний отчёт/интеграцию.

## 11. Очереди И Восстановление

Проект уже защищает складскую работу через локальные очереди:

- `pending_saves` - сканы, которые не записались в Google;
- `pending_prints` - печать, которую надо повторить;
- `pending_telegram` - файлы/сообщения, которые надо повторно отправить;
- `scan_backups` - локальная страховка каждого принятого КИЗа.

Принцип:

- КИЗ не должен считаться принятым, если не удалось записать локальный backup;
- Google/Telegram/печать могут падать, но складская работа должна восстановиться;
- оператор должен видеть понятную ошибку и следующий шаг.

Будущее: очереди должны стать централизованными в backend/PostgreSQL, чтобы несколько ПК видели одно состояние.

## 12. Автообновление

Текущая версия в коде: `APP_VERSION = 1.1.17`.

Механизм:

- публичный `version.json` в GitHub;
- GitHub Release с Windows ZIP;
- SHA256 проверка скачанного файла;
- onedir-сборка как основной формат;
- onefile exe как переходный формат для старых установок.

Решённые проблемы:

- updater больше не должен молча запускаться без подтверждения;
- есть cooldown 1 час на повторную попытку;
- при ошибке updater не перезапускает старый exe;
- `maybe_rename_windows_executable` не должен запускать старый exe в ветке ошибки;
- логи обновления пишутся в `docs/TakSklad_update.log`.

Временное состояние: `version.json` может быть переключён на `onefile_exe`, чтобы остановить цикл у старого клиента. После следующего стабильного релиза можно вернуть нормальный `onedir_zip` manifest.

## 14. Известные Общие Проблемы

1. SkladBot иногда не находит номера.
   Причины: окно по датам, пунктуация клиента, неполный набор товаров, адрес, оплата, несколько кандидатов.
   Решение: диагностика причин, 14-дневное окно, нормализация клиента, partial-match.

2. SkladBot может быть тяжёлым.
   Причины: для точной даты нужна карточка заявки, а не только список.
   Решение: окно по `created_at`, лимиты, логирование, отдельный worker в будущем.

3. Google Sheets quota `429`.
   Причины: несколько ПК, Telegram lock/state, частые `get_all_values`, SkladBot sync.
   Решение: backoff, cache, меньше чтений, backend/PostgreSQL.

4. Два ПК работают неидеально.
   Причины: локальные очереди, общий Google, Telegram внутри desktop.
   Решение: временно lock/state; долгосрочно backend и центральные очереди.

5. “Приложение занято”.
   Причины: сетевые операции, Tkinter callbacks, старые фоновые задачи.
   Решение: уже добавлен busy-state; дальше нужны timeouts, cancellation и ещё меньше сетевой работы в UI-контексте.

6. Автообновление требует осторожности.
   Причины: старые exe, package transition, Windows locks, антивирус.
   Решение: подтверждение, cooldown, понятный update log, стабильный релиз.

7. Документация местами расходится по датам и версиям.
   Решение: поддерживать changelog как источник последних решений, периодически актуализировать `taksklad-full-functionality.md`.

## 15. Что Уже Сделано Хорошо

- Импорт Excel вынесен в единый поток для UI и Telegram.
- Заказы без SkladBot-номера не скрываются.
- SkladBot больше не блокирует быстрое обновление списка.
- Telegram lock уменьшает `HTTP 409 Conflict`.
- Общий `last_update_id` снижает риск двойного импорта.
- Google errors стали понятнее.
- Локальные очереди защищают от потери сканов.
- Сохранение `TakSklad_data.json` повторяется при Windows file lock.
- Auto-update защищён от циклов.
- Логи перенесены в `docs`.
- Есть тесты на важные регрессии.

## 16. Что Нельзя Делать

- Нельзя коммитить реальные credentials, Telegram token, chat_id, private key.
- Нельзя возвращать fuzzy-match клиента, который цепляет соседних клиентов.
- Нельзя выбирать случайную заявку при нескольких совпадениях.
- Нельзя блокировать сканирование ожиданием SkladBot.
- Нельзя делать Google Sheets единственным источником восстановления сканов без локального backup.
- Нельзя запускать несколько Telegram poller без координации.
- Нельзя запускать старый exe в ветке ошибки updater.
- Нельзя сбрасывать пользовательские локальные файлы при обновлении.
